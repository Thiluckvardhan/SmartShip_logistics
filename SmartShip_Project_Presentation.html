Request Path

  • Frontend targets one API base URL through gateway.
  • Gateway routes by path prefix to internal services.
  • Services run independently and can scale separately.

Async Path

  • Business events are published after key state changes.
  • Tracking and admin contexts consume and react asynchronously.
  • Event contracts are shared through SmartShip.Contracts.

4. Event-Driven Flow

How shipment lifecycle actions propagate without tight service coupling.

Step 1 - Shipment action persists state A shipment create/book/update action is committed in ShipmentService.
Step 2 - Integration event is published Event is serialized and published through RabbitMQ with retry policy.
Step 3 - Tracking and admin contexts consume TrackingService updates timeline/location; admin context consumes for operational visibility.
Step 4 - Frontend reflects latest state Customers and admins query updated APIs through the gateway.
ShipmentCreatedEvent ShipmentBookedEvent ShipmentDeliveredEvent TrackingUpdatedEvent ShipmentIssueReportedEvent UserRegisteredEvent
ShipmentService -> RabbitMQ exchange (smartship_events) RabbitMQ -> TrackingEventConsumer (TrackingService) RabbitMQ -> Admin event handlers Tracking/Admin APIs -> Gateway -> Angular UI

7. Security and Reliability

Defense layers applied from identity flows through service communication and logs.

Authentication

  • JWT bearer token issuance and validation.
  • OTP verification as part of login flow.
  • Role-based authorization for admin endpoints.

Service Communication

  • Gateway centralizes external exposure.
  • Service-to-service calls rely on trusted tokens.
  • CORS and API boundaries managed at API layer.

Operational Resilience

  • Serilog rolling logs for each service.
  • Correlation IDs across request paths.
  • Retry patterns in event publish/consume flows.
Auth flow: Register/Login -> OTP verify -> JWT issued -> Angular interceptor adds Bearer token Observability flow: Request -> Correlation ID -> Service logs -> Daily rolling log files

8. Design Patterns in Use

Architecture and coding patterns that keep the solution scalable and maintainable.

Microservices + Bounded Contexts

Each domain owns its API, persistence, and deployment unit.

Gateway Pattern

Central route orchestration and API entry management through YARP.

Repository + Service Layers

Business logic separated from data access for cleaner unit testing.

Event-Driven Integration

Asynchronous contracts reduce direct runtime coupling.

Dependency Injection

Interface-first design across infrastructure and domain services.

DTO Contract Segregation

API DTOs protect domain entities and keep external contracts explicit.

Frontend Startup

cd Frontend npm install ng serve API base expected: https://localhost:8000

Known Notes to Mention During Presentation

  • Frontend production build may fail due to CSS budget limits; dev build succeeds for validation.
  • Identity build can fail with file lock when service process is already running.
  • Tracking consumer requires exact table names TrackingLogs and TrackingLocations.
  • Centralized logs are available in Backend/logs with daily rolling service files.
